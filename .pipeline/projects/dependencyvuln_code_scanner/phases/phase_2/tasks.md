# Phase 2 Tasks

- [ ] Task 1: Implement new ecosystem parsers (Maven, Cargo, Go, Podfile)
  - What: Build four new dependency parsers that follow the existing DependencyParser interface. MavenParser parses pom.xml (XML) extracting `<dependency>` blocks with groupId, artifactId, and version. CargoParser parses Cargo.toml (TOML) extracting `[dependencies]` and `[dev-dependencies]` tables. GoParser parses go.mod extracting `require` blocks with module path and version. PodfileParser parses CocoaPods Podfile extracting `pod` declarations with version constraints.
  - Files: depvuln/parsers/maven_parser.py (new), depvuln/parsers/cargo_parser.py (new), depvuln/parsers/go_parser.py (new), depvuln/parsers/podfile_parser.py (new), depvuln/parsers/__init__.py (modify — export new parsers), depvuln/cli.py (modify — add file-name discovery for pom.xml, Cargo.toml, go.mod, Podfile)
  - Done when: Each parser correctly extracts dependencies from a sample file of its ecosystem. MavenParser parses pom.xml with groupId:artifactId:version. CargoParser parses Cargo.toml with package name and version. GoParser parses go.mod with module path and version. PodfileParser parses Podfile with pod name and version. All parsers return empty list for missing/malformed files. Unit tests added in tests/test_parsers.py for each new parser. CLI discovers and routes pom.xml, Cargo.toml, go.mod, and Podfile to the correct parser.

- [ ] Task 2: Integrate NVD API and build CVE data merger
  - What: Implement NvdFetcher that queries the NVD API (https://services.nvd.org.nvd/1.0/json) for CVE data, with its own SQLite cache. Implement CveDataMerger that takes results from both OSV and NVD fetchers, deduplicates by CVE ID, and merges fields (preferring NVD for authoritative data, OSV for additional metadata). Update the CVE fetcher layer to use both sources.
  - Files: depvuln/cve/nvd_fetcher.py (new), depvuln/cve/merger.py (new), depvuln/cve/__init__.py (modify — export new classes), depvuln/cve/fetcher.py (modify — add NVD as secondary source), depvuln/cve/cache.py (modify — make cache reusable or parameterized for dual sources)
  - Done when: NvdFetcher correctly queries NVD API and returns CVE records with id, cvss, description, and fix fields. CveDataMerger deduplicates findings by CVE ID, merging OSV and NVD data without losing unique fields. When both sources have the same CVE, NVD data takes precedence for cvss/description. Unit tests verify deduplication (same CVE from both sources yields one result) and field merging (unique OSV fields preserved).

- [ ] Task 3: Enhance vulnerability scorer with exploit/age/depth weighting
  - What: Extend VulnScorer to incorporate additional risk factors beyond raw CVSS score. Add CVSS v3.1 vector string parsing to extract EP (Exploitability) and RL (Remediation Level) metrics. Add exploit availability weighting (CVSS v3.1 E: H/M/U → multiplier). Add package age decay (newer packages = higher risk via a configurable half-life formula). Add transitive dependency depth weighting (deeper transitive deps = lower risk). Sort order remains CRITICAL > HIGH > MEDIUM > LOW but with refined scores within each severity band.
  - Files: depvuln/scorer.py (modify — add vector parsing, exploit weighting, age decay, depth weighting), depvuln/cve/fetcher.py (modify — ensure vector strings are passed through), depvuln/cli.py (modify — add --depth option for scan command)
  - Done when: Scorer correctly parses CVSS v3.1 vector strings (e.g., "CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H"). Exploit availability (E:H) increases score, unproven exploits (E:U) decreases it. Package age decay applies a configurable decay factor (default half-life 365 days). Transitive depth weighting reduces score for deeper deps. Unit tests verify each weighting component independently and in combination. Sorting still produces CRITICAL > HIGH > MEDIUM > LOW order.

- [ ] Task 4: Implement Markdown, HTML, and SARIF report generators
  - What: Build three new report generators following the existing generator pattern. MarkdownReportGenerator produces a Markdown document with severity-colored tables (using emoji or text badges), CVE links, and summary statistics. HtmlReportGenerator produces a self-contained HTML document with inline CSS, severity-colored rows, clickable CVE links, and a summary dashboard section.   - Files: depvuln/reports/markdown_report.py (new), depvuln/reports/html_report.py (new), depvuln/reports/sarif_report.py (new), depvuln/reports/__init__.py (modify — export new generators)
  - Done when: MarkdownReportGenerator produces valid Markdown with severity headers, a table of findings with CVE links, and a summary section (total deps, vulnerable count, severity breakdown). HtmlReportGenerator produces valid HTML with inline CSS, severity-colored rows (red/orange/yellow/green), clickable links to NVD/NVD pages, and a summary dashboard. SarifReportGenerator produces valid SARIF 2.1.0 JSON that passes sarif-lint validation (or structural validation via schema). Each generator handles empty findings gracefully. Unit tests verify output structure and content.
