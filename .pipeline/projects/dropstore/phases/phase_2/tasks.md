# Phase 2 Tasks

- [ ] Task 1: Supplier API Integration Foundation
  - What: Create the supplier abstraction layer and implement adapters for at least 2 supplier sources (AliExpress/DSers and CJDropshipping). Define the base supplier interface, implement API clients for each supplier, and add supplier configuration to the app settings.
  - Files: Create `backend/services/supplier_base.py` (abstract base class with `fetch_products`, `get_product_detail`, `check_inventory`, `get_pricing` methods), `backend/services/aliexpress_supplier.py` (AliExpress/DSers API adapter), `backend/services/cjdropshipping_supplier.py` (CJDropshipping API adapter), `backend/models/supplier.py` (SupplierConnection, SupplierProduct ORM models), `shared/schemas.py` (add SupplierConnectionSchema, SupplierProductSchema, SupplierImportRequest schemas), `backend/config.py` (add supplier API key/secret settings for each supplier)
  - Done when: The supplier base class defines a consistent interface. Both AliExpress and CJDropshipping adapters implement the interface and can fetch product listings when given valid API credentials. New supplier schemas are importable from `shared/schemas.py`. Running the app with mock supplier data returns valid product listings.

- [ ] Task 2: Automated Product Import from Suppliers
  - What: Build the product import service that queries supplier APIs and imports products into the user's catalog. Handles product detail fetching, variant mapping, image/title optimization, and margin calculation from landed cost. Exposes an API endpoint for one-click import.
  - Files: Create `backend/services/supplier_import_service.py` (import logic: query supplier, map variants, compute margins, persist to catalog), `backend/routers/supplier_router.py` (API endpoints: GET /suppliers for listing connections, POST /suppliers/connect for adding credentials, POST /suppliers/{supplier_id}/import for triggering import, GET /suppliers/{supplier_id}/products for browsing supplier products), `backend/models/supplier.py` (extend with SupplierConnection model), `shared/schemas.py` (add SupplierProductDetail, ImportResult schemas), `frontend/app/suppliers/page.tsx` (supplier connection and product browsing UI), `frontend/components/SupplierProductCard.tsx` (product card for supplier browsing), `frontend/components/SupplierImportWizard.tsx` (step-by-step import flow)
  - Done when: A user can connect to ≥2 supplier sources (AliExpress and CJDropshipping), browse their product listings, and import products into their catalog with one click. Imported products include correct title, image, variants, cost, and margin data. The import API returns a result showing products imported, skipped, and failed.

- [ ] Task 3: Price & Inventory Sync Engine
  - What: Build the scheduled sync engine that periodically queries supplier APIs for price and inventory updates, applies changes to catalog products, and logs sync results. Uses Redis-based scheduling with configurable intervals (default: every 6 hours). Includes retry logic for failed syncs.
  - Files: Create `backend/services/sync_engine.py` (core sync engine: scheduled price/inventory fetch, diff detection, catalog update), `backend/services/sync_scheduler.py` (Redis-based cron/scheduler with configurable intervals), `backend/models/sync_log.py` (SyncLog ORM model for tracking each sync run), `backend/routers/sync_router.py` (API endpoints: GET /sync/config, PUT /sync/config for schedule settings, GET /sync/logs for history, POST /sync/run for manual trigger, POST /sync/logs/{log_id}/retry for retrying failed syncs), `shared/schemas.py` (add SyncConfig, SyncLog schemas), `backend/utils/database.py` (add sync_log table migration), `frontend/app/sync/page.tsx` (sync config and history UI), `frontend/components/SyncConfigPanel.tsx`, `frontend/components/SyncLogTable.tsx`
  - Done when: Price and inventory updates run automatically on a configurable schedule (default: every 6 hours). Sync results are logged with success/failure details. Failed syncs can be retried individually or in bulk without data loss. Manual sync trigger works immediately. Sync logs show timestamp, products updated, products failed, and error messages.

- [ ] Task 4: Margin Optimizer
  - What: Build a rule-based pricing engine that allows users to define margin rules (e.g., "always markup 40% over landed cost", "minimum margin 30%", "round prices to .99"). The optimizer applies these rules to catalog products and can recompute all margins in bulk.
  - Files: Create `backend/services/margin_optimizer.py` (rule engine: parse margin rules, apply to products, handle edge cases like negative margins), `backend/models/margin_rule.py` (MarginRule ORM model), `backend/routers/margin_router.py` (API endpoints: GET /margin/rules, POST /margin/rules for creating rules, PUT /margin/rules/{rule_id} for updating, DELETE /margin/rules/{rule_id}, POST /margin/apply for applying rules to catalog), `shared/schemas.py` (add MarginRuleSchema, MarginApplyRequest, MarginApplyResult schemas), `backend/utils/database.py` (add margin_rules table migration), `frontend/app/margins/page.tsx` (margin rules management UI), `frontend/components/MarginRuleEditor.tsx`, `frontend/components/MarginApplyDialog.tsx`
  - Done when: Users can create, edit, and delete margin rules with configurable markup percentages, minimum margins, and price rounding. Applying rules to a catalog recomputes all product margins accurately within 2% of the configured rule. The optimizer handles edge cases (e.g., zero-cost products, negative margins) gracefully. Applied margins persist to the database.

- [ ] Task 5: Alert System
  - Build the notification system that monitors supplier events and alerts users to critical issues: low stock, price changes exceeding thresholds, and supplier connection failures. Includes alert configuration (thresholds, channels) and a notification center in the dashboard.
  - Files: Create `backend/services/alert_service.py` (alert generation, filtering, and dispatch), `backend/services/alert_monitor.py` (background monitor that checks sync results and triggers alerts), `backend/models/alert.py` (Alert ORM model with type, severity, status fields), `backend/routers/alert_router.py` (API endpoints: GET /alerts for listing, GET /alerts/{alert_id} for detail, PUT /alerts/{alert_id}/dismiss for dismissing, GET /alert-config for settings, PUT /alert-config for updating thresholds), `shared/schemas.py` (add AlertSchema, AlertConfigSchema, AlertThreshold schemas), `backend/utils/database.py` (add alerts table migration), `frontend/app/alerts/page.tsx` (alert center UI), `frontend/components/AlertFeed.tsx`, `frontend/components/AlertConfigPanel.tsx`, `frontend/components/AlertBadge.tsx` (dashboard badge for unread alerts)
  - Done when: Users receive alerts for ≥95% of critical supplier events (low stock, price changes > configured threshold, supplier connection failures). Alerts include product name, supplier, severity, and action suggestions. Users can configure alert thresholds per product or globally. Dismissed alerts are hidden but remain in history. The dashboard shows an unread alert count badge.
