# Phase 3 Tasks

- [ ] Task 1: Multi-Platform Store Support (WooCommerce)
  - What: Extend the Store model to support multiple platform types and implement a WooCommerce adapter for product/order sync. Create a unified sync interface that works across Shopify and WooCommerce, extending the existing sync engine to dispatch to the correct platform adapter based on store type.
  - Files: Create/modify `backend/models/store.py` (extend Store model with `platform_type` enum: 'shopify', 'woocommerce', 'custom_headless', and platform-specific fields like `woo_url`, `woo_consumer_key`, `woo_consumer_secret`), `backend/services/store_base.py` (abstract base class for store adapters with `sync_products`, `fetch_orders`, `fetch_inventory`, `push_product` methods), `backend/services/woocommerce_store.py` (WooCommerce REST API adapter implementing the store base interface), `backend/services/sync_engine.py` (extend to dispatch sync jobs to the correct platform adapter), `backend/routers/store_router.py` (add POST /stores/connect/woocommerce, GET /stores, PUT /stores/{id} endpoints), `shared/schemas.py` (add WooCommerceStoreSchema, StoreConnectionRequest schemas), `backend/utils/database.py` (add store platform_type column and WooCommerce-specific columns), `frontend/app/connect/page.tsx` (add WooCommerce connection flow), `frontend/components/WooCommerceConnect.tsx` (WooCommerce OAuth/API key connection UI)
  - Done when: A user can connect a WooCommerce store by providing their store URL and API keys. Products from the Dropstore catalog can be synced to the WooCommerce store (title, description, images, price, variants). Orders from WooCommerce can be fetched back into Dropstore. The sync engine correctly routes sync jobs to the appropriate platform adapter. Both Shopify and WooCommerce stores can coexist for a single user.

- [ ] Task 2: Store Templates System
  - What: Build a template system that lets users deploy pre-built store themes/layouts for common niches (fashion, electronics, home goods, pet supplies, fitness, etc.). Each template defines CSS styles, layout components, color palettes, and default store settings. Include a template gallery UI and one-click deployment.
  - Files: Create `backend/models/store_template.py` (StoreTemplate, TemplateSection, TemplateStyle ORM models), `backend/services/template_service.py` (template registry, validation, deployment logic), `backend/routers/template_router.py` (API endpoints: GET /templates for gallery, GET /templates/{id} for preview, POST /templates/{id}/deploy for deployment), `shared/schemas.py` (add StoreTemplateSchema, TemplateDeployRequest schemas), `frontend/app/templates/page.tsx` (template gallery with preview cards), `frontend/components/TemplatePreview.tsx` (live preview of template styles), `frontend/components/TemplateDeployDialog.tsx` (deployment confirmation and progress), `frontend/app/store/[storeId]/styles/` (template CSS/style injection service), `backend/services/style_injector.py` (applies template styles to the connected storefront via API or CSS injection)
  - Done when: ≥6 store templates are available covering common niches (fashion, electronics, home goods, pet supplies, fitness, beauty). A user can preview any template and deploy it to a connected store in <10 minutes. Deployed templates apply correct color palettes, layouts, typography, and product grid styles. Templates can be swapped or reverted at any time. Template styles are applied via the platform's API (Shopify theme API / WooCommerce customizer API).

- [ ] Task 3: Unified Order Management
  - What: Build a centralized order management system that tracks orders from all connected stores (Shopify, WooCommerce) and maps them to supplier orders. Include order status tracking, fulfillment updates, and a unified order list in the dashboard.
  - Files: Create `backend/models/order.py` (Order, OrderItem, OrderFulfillment ORM models with store_id, supplier_order_id, status, tracking info, profit), `backend/services/order_service.py` (order creation, status updates, fulfillment tracking, profit calculation), `backend/services/order_sync.py` (fetch orders from Shopify and WooCommerce APIs, normalize to canonical Order model, handle webhooks for real-time updates), `backend/routers/order_router.py` (API endpoints: GET /orders for listing, GET /orders/{id} for detail, PUT /orders/{id}/status for manual updates, GET /orders/export for CSV export, POST /orders/{id}/fulfill for fulfillment), `shared/schemas.py` (add OrderSchema, OrderItemSchema, OrderFulfillmentSchema, OrderFilter schemas), `backend/utils/database.py` (add orders, order_items, order_fulfillments table migrations), `frontend/app/orders/page.tsx` (order management dashboard), `frontend/components/OrderList.tsx` (filterable, sortable order table), `frontend/components/OrderDetailPanel.tsx` (order detail with status timeline), `frontend/components/OrderStatusBadge.tsx`
  - Done when: Orders from ≥2 platforms (Shopify and WooCommerce) appear in a unified order list with correct status, product details, and profit data. Real-time webhook updates reflect status changes within 5 minutes. Users can manually update order status and add tracking information. Order export to CSV works correctly. ≥95% of orders from connected stores are captured and displayed accurately.

- [ ] Task 4: Analytics Dashboard
  - What: Build an analytics dashboard that tracks ≥5 key metrics: total revenue, total orders, average order value, conversion rate, top products by revenue, and profit margin trends. Include time-series charts and product performance rankings.
  - Files: Create `backend/models/analytics.py` (DailyMetrics, ProductPerformance, RevenueSummary ORM models for aggregated analytics data), `backend/services/analytics_service.py` (metrics aggregation: revenue, orders, AOV, conversion rate, top products, margin trends; time-series computation), `backend/services/analytics_aggregator.py` (scheduled job to aggregate raw order/product data into metrics), `backend/routers/analytics_router.py` (API endpoints: GET /analytics/overview for summary metrics, GET /analytics/revenue for time-series, GET /analytics/products for top products, GET /analytics/conversion for conversion metrics, GET /analytics/margins for margin trends), `shared/schemas.py` (add AnalyticsOverviewSchema, RevenueTimeSeriesSchema, ProductPerformanceSchema schemas), `backend/utils/database.py` (add analytics tables and aggregation migrations), `frontend/app/analytics/page.tsx` (analytics dashboard), `frontend/components/RevenueChart.tsx` (revenue time-series chart), `frontend/components/OrderMetricsCards.tsx` (AOV, conversion rate, total orders cards), `frontend/components/TopProductsTable.tsx` (product performance ranking), `frontend/components/MarginTrendChart.tsx`
  - Done when: The analytics dashboard displays ≥5 key metrics (total revenue, total orders, average order value, conversion rate, top products by revenue, profit margin trends) with accurate data derived from connected store orders. Time-series charts show daily/weekly/monthly revenue trends. Product performance rankings show revenue and margin per product. Metrics update within 1 hour of new order data. All metrics are independently verifiable against source store data.

- [ ] Task 5: Automated Marketing Tools
  - What: Build automated content generation for marketing: SEO-optimized product descriptions, social media post templates, and email campaign templates. Use the product catalog data to generate platform-specific marketing content that users can review and publish.
  - Files: Create `backend/services/marketing_generator.py` (content generation engine: SEO descriptions, social media posts, email templates using product data), `backend/services/seo_optimizer.py` (SEO-specific: keyword extraction, meta description generation, alt-text generation), `backend/services/social_generator.py` (social media: platform-specific post formats for Instagram, Facebook, Twitter/X, Pinterest), `backend/services/email_generator.py` (email campaigns: product launch emails, abandoned cart emails, promotional emails), `backend/routers/marketing_router.py` (API endpoints: GET /marketing/products/{id}/seo for SEO content, POST /marketing/generate for bulk generation, GET /marketing/campaigns for templates, POST /marketing/campaigns for creating campaigns), `shared/schemas.py` (add SEODescriptionSchema, SocialMediaPostSchema, EmailCampaignSchema schemas), `frontend/app/marketing/page.tsx` (marketing tools dashboard), `frontend/components/SEODescriptionEditor.tsx` (editable SEO content), `frontend/components/SocialPostPreview.tsx` (social media post preview), `frontend/components/EmailCampaignBuilder.tsx` (email template builder), `frontend/components/MarketingContentList.tsx`
  - Done when: For each catalog product, the system generates an SEO-optimized description (with keywords, meta description, alt-text), ≥3 social media post variants (Instagram, Facebook, Twitter/X), and ≥2 email campaign templates. Generated content is human-reviewable and editable before publishing. Bulk generation across a full catalog works within 2 minutes. Content quality is usable without major rewrites (passes basic readability and keyword relevance checks).

- [ ] Task 6: Team & Role Management
  - What: Implement multi-user support with role-based access control (RBAC). Support three roles: admin (full access), editor (product/catalog/sync management), and viewer (read-only dashboard access). Include user invitation, authentication, and permission checks on all API endpoints.
  - Files: Create `backend/models/user.py` (User, Team, TeamMember, Role ORM models with role enum: 'admin', 'editor', 'viewer'), `backend/services/auth_service.py` (user authentication, invitation flow, token management), `backend/services/permission_service.py` (RBAC middleware: check role permissions on endpoints), `backend/middleware/permissions.py` (FastAPI dependency for permission checks), `backend/routers/user_router.py` (API endpoints: POST /users/invite for invitations, GET /users for team list, PUT /users/{id}/role for role changes, DELETE /users/{id} for removal, GET /users/me for current user), `backend/routers/` (add permission decorators to existing routers: /stores, /catalog, /orders, /analytics, /marketing), `shared/schemas.py` (add UserSchema, TeamMemberSchema, InviteRequest, RoleChange schemas), `backend/utils/database.py` (add users, teams, team_members table migrations), `frontend/app/team/page.tsx` (team management UI), `frontend/components/TeamMemberList.tsx` (invite, role change, remove), `frontend/components/RoleSelector.tsx`, `frontend/components/PermissionGuard.tsx` (UI elements hidden based on role)
  - Done when: A team owner can invite members via email and assign roles (admin, editor, viewer). Admins have full access to all features. Editors can manage products, catalogs, sync, and marketing but cannot manage team members or billing. Viewers have read-only access to dashboard and analytics. Permission checks enforce roles on all API endpoints (403 for unauthorized access). Role changes take effect immediately. Team members see appropriate UI elements based on their role.
