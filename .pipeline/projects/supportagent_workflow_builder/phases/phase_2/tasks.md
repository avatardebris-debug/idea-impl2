# Phase 2 Tasks

- [ ] Task 1: SOP DSL Engine — YAML workflow definitions, multi-step execution, conditional branching, and human-in-the-loop gates
  - What: Build the core SOP engine that reads YAML workflow definitions and executes multi-step sequences with conditional branches and human-in-the-loop approval gates. Each step in a workflow can be an action (e.g., "assign_team", "send_email", "generate_draft"), a condition (e.g., "if priority >= 8"), or a gate (e.g., "await human approval"). The engine maintains execution state so workflows can be paused at gates and resumed later without data loss.
  - Files: 
    - Create `workspace/supportagent/sop_engine.py` — `SOPWorkflow` model, `SOPStep`, `SOPCondition`, `SOPGate`, `WorkflowEngine` classes
    - Create `workspace/supportagent/sop_engine.py` — `WorkflowState` dataclass for pause/resume state
    - Create `workspace/supportagent/config/sop_workflows/` directory with 5 sample YAML workflow files (billing_flow.yaml, technical_flow.yaml, account_flow.yaml, escalation_flow.yaml, general_flow.yaml)
    - Modify `workspace/supportagent/models.py` — add `WorkflowState` and `SOPStepResult` to the model exports
  - Done when: 
    - Engine can load and execute 5 distinct multi-step YAML workflows end-to-end
    - Conditional branches correctly route execution based on ticket attributes (priority, category, sentiment)
    - Human-in-the-loop gates pause execution, persist state to a `WorkflowState` object, and resume without data loss
    - Unit tests cover: workflow loading, step execution, condition evaluation, gate pause/resume, escalation trigger, and error handling (≥ 20 tests)

- [ ] Task 2: Workflow Versioning & Rollback System
  - What: Add version management for SOP workflows. Each workflow definition is stored with a version number. The system supports creating new versions, listing versions, and rolling back to any prior version. Version metadata includes timestamp, author, and change summary. Rollback atomically replaces the active workflow with a previous version.
  - Files:
    - Create `workspace/supportagent/sop_versioning.py` — `WorkflowVersion`, `VersionStore`, `VersionManager` classes
    - Create `workspace/supportagent/config/sop_versions/` directory for versioned workflow storage
    - Modify `workspace/supportagent/sop_engine.py` — integrate version lookup so the engine loads the active version by default
    - Add CLI commands in `workspace/supportagent/cli.py` for version management (list, rollback, diff)
  - Done when:
    - Each workflow supports ≥ 3 versions with full metadata (timestamp, author, change summary)
    - Rollback atomically switches the active workflow version and the engine picks up the new version on next load
    - Version diff shows step-by-step changes between versions
    - Unit tests cover: version creation, listing, rollback, diff generation, and concurrent version safety (≥ 10 tests)

- [ ] Task 3: Enhanced Triage Engine — Sentiment analysis, priority scoring, and ML-assisted classification
  - What: Extend the existing `Classifier` with three capabilities: (1) Sentiment analysis that classifies ticket text as positive / neutral / negative / angry using a lexicon-based approach (no external ML dependency required for MVP, but architecture supports swapping in an ML model later). (2) Priority scoring on a 1–5 scale based on urgency signals (keywords, sentiment, customer tier, SLA status). (3) ML-assisted auto-categorization using a zero-shot classification approach (e.g., using sentence embeddings + cosine similarity to category prototypes, or a lightweight transformer if available).
  - Files:
    - Create `workspace/supportagent/enhanced_triage.py` — `SentimentAnalyzer`, `PriorityScorer`, `MLClassifier` classes
    - Create `workspace/supportagent/config/sentiment_lexicon.yaml` — word lists for positive, neutral, negative, angry sentiment
    - Create `workspace/supportagent/config/ml_category_prototypes.yaml` — category prototype vectors for zero-shot classification
    - Modify `workspace/supportagent/classifier.py` — integrate enhanced triage as a pipeline step; add `EnhancedClassifier` wrapper that calls sentiment, priority, and ML classifier in sequence
    - Modify `workspace/supportagent/models.py` — add `Sentiment` enum and update `Ticket` to include enhanced triage fields
  - Done when:
    - Sentiment analyzer correctly classifies ≥ 80% of test tickets (F1 ≥ 0.80 on held-out test set of 100 labeled tickets)
    - Priority scorer produces 1–5 scores that correlate with urgency signals (verified via test assertions)
    - ML classifier achieves ≥ 90% accuracy on a held-out test set of 200 labeled tickets
    - Unit tests cover: sentiment analysis (all 4 classes), priority scoring edge cases, ML classification accuracy, and integration with existing Classifier (≥ 25 tests)

- [ ] Task 4: LLM-Assisted Response Generator — Template fallback, LLM generation, and tone/style customization
  - What: Build a response generator that first attempts template-based matching for known ticket patterns (using Jinja2 templates from `supportagent/config/templates/`). For unknown or novel ticket types, it falls back to LLM-assisted generation with confidence scoring. Supports tone/style customization per team or customer tier (e.g., formal for enterprise, casual for startup). The generator produces draft responses with metadata including confidence score, generation method (template vs. LLM), and applicable tone.
  - Files:
    - Create `workspace/supportagent/response_generator.py` — `TemplateResponseGenerator`, `LLMResponseGenerator`, `ResponseGenerator` (orchestrator) classes
    - Create `workspace/supportagent/config/response_templates/` directory with ≥ 5 Jinja2 templates (billing, technical, account, urgent, general)
    - Create `workspace/supportagent/config/tone_styles.yaml` — tone/style definitions per team and customer tier
    - Modify `workspace/supportagent/models.py` — add `ResponseMetadata` dataclass with confidence_score, generation_method, tone fields
    - Modify `workspace/supportagent/router.py` — integrate response generation into the routing pipeline
  - Done when:
    - Template matching correctly generates responses for all 5 template types with variable substitution
    - LLM fallback generates acceptable responses for unmatched ticket types with confidence scoring
    - Tone customization works per team and customer tier (verified via test assertions on output text)
    - Unit tests cover: template rendering, LLM fallback simulation, tone customization, confidence scoring, and pipeline integration (≥ 20 tests)
