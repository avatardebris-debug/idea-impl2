The Core Insight
Your current autonomous pipeline is structurally identical to the robot training loop you're describing. You're not building toward that world someday — you're already building the foundation.

CURRENT PIPELINE                    ROBOT TRAINING LOOP
─────────────────────────────────   ─────────────────────────────────
executor agent                  ≡   robot attempts task
pytest validation               ≡   physics engine evaluates outcome
reviewer agent                  ≡   critic scores sim output
manager agent                   ≡   goal decomposer / curriculum planner
retry / fix loop                ≡   RL improvement iteration
pipeline runner                 ≡   training orchestrator
The gap between "autonomous software builder" and "autonomous robot trainer" is mostly: (a) swapping Python code generation for MuJoCo program generation, and (b) swapping pytest with physics simulation scoring.

What Can Be Built Today (No A100 Required)
Tier 1: Pure LLM + Python (pipeline can build these)
video_recipe — LLM takes video_scribe output (structured scene descriptions) and extracts an ordered list of atomic actions:

json
[
  {"step": 1, "action": "grasp", "object": "hammer",
   "xyz_delta": [0.0, 0.0, -0.15], "duration_s": 0.8,
   "preconditions": ["hammer_on_surface", "gripper_open"]},
  {"step": 2, "action": "swing", "object": "hammer",
   "direction": "down", "force_estimate": "heavy",
   "target": "nail", "xyz_delta": [0.0, 0.0, -0.3]}
]
This is 100% LLM work. No GPU, no physics engine.

robo_primitive_mapper — Maps the above recipe to abstract robot commands. Define a primitive set (similar to MuJoCo's built-ins):

move_to(xyz, frame="world|gripper|object")
grasp(object_id, grip_force)
release()
apply_force(direction, magnitude, duration)
wait(seconds)
observe(sensor="camera|force|joint_pos")
This is a translation problem — LLM with structured output. The key design decision is the primitive vocabulary: define it now and lock it in. Everything downstream builds on it.

sim_comparator (perceptual only) — Compare two video clips using:

SSIM (structural similarity — OpenCV, no GPU)
Perceptual hash (difference hash — 16-bit, CPU only)
CLIP embeddings (if GPU available, else skip)
Output a [0,1] similarity score + frame-by-frame heatmap
robot_skill_library — SQLite + FAISS/ChromaDB vector store. Skills indexed by: name, description embedding, input/output object types. Query: "find skills similar to 'hammer a nail'".

goal_decomposer — LLM agent with tool access:

search_skill_library(query) → returns matching skills
add_skill_gap(skill_description) → queues video_recipe to find a demo
compose_plan(goal, skills) → builds dependency tree of skills
Tier 2: Needs A100-class GPU (run on VastAI now, local in ~5 years)
mujoco_codegen — Generates MuJoCo XML + Python control script. Requires testing against an actual physics simulation. VastAI can run this.

sim_real_discriminator — Extends video_GAN with:

Training data: (sim_frames, real_frames) pairs
Loss: binary cross-entropy (real=1, sim=0)
Output: per-frame "realness" score + global score
Needs at least a few hundred training pairs — robot camera footage vs renders
Tier 3: Requires Real Hardware + Datacenter
Actual sim-to-real gap closure
Physical robot in training facility
Real camera footage collection
Policy distillation from sim to real weights
Why the Discriminator is the Key Investment
The sim_real_discriminator is the linchpin because:

You can start training it now on synthetic data — generate pairs of (UE5/MuJoCo render, real video of the same action) and train the classifier. Your video_GAN already has this architecture.

Once trained, it becomes a universal reward signal — any new skill attempt gets a "realness score" without needing ground truth labels.

It self-improves with the generator — as MuJoCo renders get more realistic (better materials, physics params), the discriminator gets harder to fool, driving the generator to improve further.

It's the bridge between now and the robot — even before you have a physical robot, you can improve the discriminator using YouTube footage of humans and robots performing tasks.

The Primitive Vocabulary — Most Important Decision to Make Now
The primitive set should be:

Small (<30 primitives) — discoverable, composable
Complete — enough to describe most manipulation tasks
Grounded — each has a clear physical interpretation in XYZ space
Proposed set:

LOCOMOTION:    move_to, rotate_to, approach, retreat
MANIPULATION:  grasp, release, push, pull, lift, place, insert, rotate_object
OBSERVATION:   look_at, scan, measure_distance, detect_object
FORCE:         apply_force, apply_torque, maintain_contact
COMPOUND:      sequence, parallel, repeat_until, conditional
STATE:         wait, pause, signal_done, request_human
This is a vocabulary design problem that benefits enormously from human input (like this conversation) rather than LLM generation — the LLM will respect and reuse whatever vocabulary we define.

Core Structural Insight: One Pipeline, Multiple Domains
Not a fork. A mode.

The question of "fork vs. unified" is decided by one scenario:

/goals "build a house and list it on Airbnb"
Subgoal tree:
  ├── acquire building permit          [domain: software → legal_doc_generator]
  ├── dig foundation                   [domain: robot   → mujoco_codegen]
  ├── frame walls                      [domain: robot   → mujoco_codegen]
  ├── source materials online          [domain: software → web_scraper + purchasing]

  ├── photograph completed house       [domain: robot   → camera_control]
  ├── write Airbnb listing             [domain: software → ai_author_suite]

A forked runner needs IPC between processes to coordinate this. A unified runner with a domain field handles it in one goal graph natively. The fork is only the prompt templates and tool wrappers — not the orchestration system.

PufferLib — The Compute Efficiency Multiplier
PufferLib (github.com/PufferAI/PufferLib) achieves 10-100x sample efficiency vs. naive RL implementations through:

Vectorized environments — 512+ parallel MuJoCo instances per GPU
Async rollout collection — GPU never idles waiting for env steps
Clean env/policy separation — easy to plug in MuJoCo or custom envs
Battle-tested — used in NeurIPS competition settings
What this changes for the hardware roadmap:

Task	Naive RL	With PufferLib	Hardware
Train "grasp object"	~1 week, A100	~4 hours, RTX 4090	Now
Train "stack 3 blocks"	~2 weeks, 4×A100	~1 day, RTX 4090	Now
Train "assemble chair"	months, cluster	~1 week, 2×RTX 4090	$1k-2k setup
Full house build skill tree	years, datacenter	months, 8×RTX 4090	~5 years at $1k/A100
The discriminator score from sim_real_discriminator feeds directly into PufferLib as the reward function — no custom RL infrastructure needed.
